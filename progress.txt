## Codebase Patterns
- Use Drizzle ORM with `drizzle-orm/libsql` for Turso database access
- SQLite stores JSON as text - use `text('field', { mode: 'json' }).$type<T>()` for type-safe JSON columns
- Use `crypto.randomUUID()` for generating primary keys in schema defaults
- Drizzle config file should be excluded from tsconfig.json to avoid type conflicts with the CLI tool
- Environment variables for Turso: TURSO_DATABASE_URL and TURSO_AUTH_TOKEN must be set
- All database tables cascade delete when user is deleted (foreign key constraint)
- NextAuth v5: Configuration lives in `src/auth.ts`, export `auth()` for server-side session access
- NextAuth adapter requires exact schema: sessions table uses sessionToken as PK, verificationTokens has composite PK
- Use `auth()` from src/auth.ts server-side, `useSession()` from next-auth/react client-side
- Protected routes: Middleware redirects pages to sign-in, returns 401 for API routes
- Data transform pattern: DB schema types differ from app types - create conversion functions (e.g., dbSiteToAppSite)
- Always scope queries to authenticated user: `eq(table.userId, session.user.id)` in where clauses
- Use `and()` from drizzle-orm to combine multiple where conditions
- Upsert pattern for singleton user data: `.insert().onConflictDoUpdate({ target: table.userId, set: {...} })`
- Upsert with composite key: `.onConflictDoUpdate({ target: [table.col1, table.col2], set: {...} })`
- Unique constraints in Drizzle: Use `unique().on(table.col1, table.col2)` in table config callback
- Generate migrations with `npx drizzle-kit generate` (auto-detects schema changes)

## 2026-03-04 21:35 - US-006
- Fixed Scoring Algorithm with circular mean and upper-air parameters
- Files changed:
  - Modified: src/types/index.ts (added blh and upperWind to DayScore.factors)
  - Modified: src/lib/scoring.ts (complete scoring overhaul)
- **Learnings for future iterations:**
  - **Circular mean for wind direction**: Wind directions are circular data (0° = 360°), so arithmetic mean fails at boundaries
    * Formula: atan2(mean(sin(dirs)), mean(cos(dirs))) converted back to degrees
    * This gives correct "center" direction (e.g., average of 10° and 350° = 0°, not 180°)
  - **Boundary layer height (BLH)**: Height of mixing layer where thermals develop (0-3000m typical range)
    * Higher BLH = deeper thermals = better XC potential
    * Scoring: 0m=0, 500m=30, 1000m=60, 1500m=80, 2000m+=100
  - **850hPa wind**: Pressure level at roughly 1500m altitude - critical for XC safety
    * Shows wind at typical thermal top height
    * Strong upper winds = drift, turbulence, dangerous conditions
    * Scoring: <20km/h=100, 20-40=70, 40-60=40, 60-80=20, >80=0
  - **Revised weights**: CAPE 25%, surface wind 20%, wind direction 15%, cloud cover 10%, precipitation 5%, BLH 15%, upper wind 10%
    * Total = 100% (must always sum to 100%)
  - **Null handling**: New parameters (BLH, 850hPa) can be null if API doesn't provide them
    * Use neutral score (50) if missing - don't penalize missing data
    * Filter null values before averaging: `.filter((v): v is number => v !== null)`
  - **Score labels unchanged**: Poor 0-30, Fair 31-50, Good 51-70, Great 71-85, Epic 86-100
  - Updated HourlyDataPoint interface to include all upper-air fields
